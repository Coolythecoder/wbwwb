# Native WBWWB C++ Port Notes

This folder is a true native C++/raylib port. It does not embed the original HTML/JavaScript game, Chromium, Electron, a webview, or a JavaScript runtime.

## Build Status

- CMake builds a Windows/Linux native executable.
- CMake exposes `WBWWB_RENDER_BACKEND`; `raylib` is the default Windows/Linux backend and `vulkan` is the playable raylib-free Windows x64 backend.
- raylib is fetched by CMake and linked statically.
- nlohmann_json is fetched by CMake for TexturePacker/PIXI-style atlas parsing.
- The executable loads the existing repo art and audio assets directly.
- Runtime asset-root discovery supports launching from the repo root, build folders, Visual Studio, and other CMake launch directories.
- Native app/window icons are generated from the game TV art. Windows builds embed `native/resources/wbwwb.ico` and also apply that embedded icon directly to the live window handle; runtime windows use multi-size icons derived from `sprites/app_icon.png`; Linux-style installs place the icon and desktop entry under `share/icons` and `share/applications`.

Validated locally:

```sh
cmake -S native -B native/build
cmake --build native/build --config Release
cmake --build native/build --config Debug
```

The default backend is `raylib`. To select the current renderer explicitly:

```sh
cmake -S native -B native/build-raylib -DWBWWB_RENDER_BACKEND=raylib
cmake --build native/build-raylib --config Release
```

Validated 32-bit Windows build:

```bat
cmake -S native -B native/build-win32 -G "Visual Studio 17 2022" -A Win32
cmake --build native/build-win32 --config Release
```

The Win32 output is `native/build-win32/Release/wbwwb_native.exe`. The helper `native/build-win32.bat` runs those two commands. A 32-bit build helps older Windows/x86 installs, but it still depends on raylib/OpenGL support from the machine's graphics driver.

Packaged 32-bit output can be created with:

```bat
cmake --install native/build-win32 --config Release --prefix native/out/wbwwb-win32
```

Run the packaged build from `native/out/wbwwb-win32/bin/wbwwb_native.exe`.

Validated Windows 64-bit Vulkan backend launch build:

```bat
cmake -S native -B native/build-vulkan -DWBWWB_RENDER_BACKEND=vulkan
cmake --build native/build-vulkan --config Release
native\build-vulkan\Release\wbwwb_native.exe
native\build-vulkan\Release\wbwwb_native.exe --smoke-test
```

The Vulkan target owns a Win32 app/window, Vulkan surface/device/swapchain, exact player-selected scene target with 960x540 authored coordinates, shared scene/gameplay renderer, matching-resolution full-screen pictures and photo targets, miniaudio playback, post-processing, GPU MSAA, and SDR/HDR presentation. A no-flag launch constructs the shared `Game`; focused smoke/UI/renderer/capture/HDR/MSAA/resolution tests remain available for regression coverage. `--vulkan-boundary-test` keeps its historical name but now enters real `GameplayScene` and exercises the real camera/TV capture path.

## Vulkan/HDR Backend Status

The migration reached its playable Windows x64 target. The Vulkan executable constructs the shared `Game` and `SceneManager`, reaches real `GameplayScene`, renders shared assets and atlas animation, handles Win32 input/display state, plays audio through miniaudio, and captures the camera/TV crop at the player-selected render resolution. It does not load or call raylib when Vulkan is selected.

The final Vulkan presentation shader implements the same visual controls as the raylib final pass. GPU MSAA uses a multisampled scene target and resolve image, with 2x/4x/8x/16x requests falling back to the highest supported level. SDR presentation uses an inactive scRGB composition surface when available so the Windows compositor receives correctly linearized colors, while retaining an SDR UNORM fallback. HDR mode selects scRGB when available or HDR10/PQ otherwise, submits Rec. 2020/D65 metadata, uses a 203-nit paper white and 1000-nit peak, and falls back to SDR when no HDR surface format is exposed.

Raylib remains the default backend, the 32-bit Windows backend, and the Linux backend. Vulkan's Win32 surface, WIC decoder, and GDI font atlas are intentionally Windows-only. The remaining work is release QA and portability rather than a shared-game startup blocker: Linux Vulkan integration, scripts beyond the current Latin font atlas, rotated TexturePacker frames if future assets introduce them, and a human full-game sweep through every late branch.

## Backend-neutral gameplay migration checklist

The service migration is complete for the Windows Vulkan target. `Game` consumes injected `AppWindow`, `InputProvider`, `CaptureService`, `AudioBackend`, `Renderer`, and `FramePipeline` objects. Raylib assembles those services in `RaylibGame.cpp`; Vulkan assembles `VulkanAppWindow`, `VulkanInputProvider`, `VulkanCaptureService`, `MiniaudioAudioBackend`, `VulkanRenderer`, and `VulkanFramePipeline` in `VulkanSmokeMain.cpp`.

Normal Vulkan startup enters shared `PreloaderScene` and `SettingsScene`; Play follows the shared quote/gameplay sequence. The retained `--vulkan-boundary-test` name now means a deterministic regression across that former boundary: it starts the shared game, enters `GameplayScene`, takes a real photo, validates non-black readback, and exercises TV presentation. There is no raylib runtime fallback in the Vulkan build.

## Raylib compatibility translation layer

`native/src/compat/RaylibCompat.h` and `.cpp` are a game-specific compatibility vocabulary, not a raylib dependency. They define neutral `wb::Vector2`, `wb::Rectangle`, and `wb::Color`, conversions to renderer types, collision/log/open-URL helpers, and renderer-backed lines, circles, outlines, and rounded rectangles. `Game` installs the same context for both backends and clears it before service teardown.

`AssetManager` exposes only renderer handles and neutral atlas metadata. Raw `Texture2D`, `RenderTexture2D`, `Font`, `Sound`, `Music`, window calls, and input polling are confined to `native/src/render/raylib/*`. Vulkan calls this compatibility layer while running real shared gameplay; it neither embeds nor silently invokes raylib.

## Screenshot And TV Behavior

The original JS camera captures `scene.world.graphics` into a 240x135 `PIXI.RenderTexture` at the camera bounds. The capture excludes the camera frame/instructions and is stored as the current photo texture. During the TV cut, the TV receives that exact photo texture, draws it at half scale inside the TV screen area, then draws the chyron container over the photo. The original `Camera.js` source has a `TO DO: higher rez` note on that render-texture capture path.

The native port keeps the same logical camera crop, timing, and TV composition, but stores the crop at the selected render resolution for quality. `CameraRig` owns a backend-neutral `RenderTargetHandle` whose dimensions track the main scene target; clicking captures the current 240x135 world-only camera view at that resolution. `Director::cutToTV()` passes its texture handle to `TV::placePhoto()`. `TV::draw()` still draws `sprites/tv.png` first, then the captured photo in the original 120x67.5 logical TV rectangle, then the chyron. Raylib and Vulkan implement the target separately. The photo is not saved to disk and excludes post-processing, OS chrome, letterbox bars, the camera frame/instructions, and the desktop.

## Native Settings

The native port adds a separate `SettingsScene` instead of changing the story/captured-screenshot TV behavior. The main menu opens this scene from a small settings control on the preloader TV; returning from settings goes back to the menu and the original play flow still enters the quote/gameplay sequence. During gameplay and the other story scenes, Escape opens a modal pause menu with Resume, Settings, and a confirmed Exit Game action. Opening settings from that menu freezes and preserves the current scene, and Back or Escape returns to the pause menu.

The settings page draws the existing hand-drawn background and `sprites/tv.png`, then clips all controls to the TV screen rectangle in the 960x540 logical coordinate space. The page now has Audio, Display, Visuals, and Monitor tabs. Long tabs use a clipped vertical scroll area; mouse wheel scrolling works inside the TV screen, keyboard navigation scrolls the selected row into view, and Back remains reachable.

Settings are saved to `wbwwb_settings.json` in the current working directory. New snake_case fields include `display_mode`, `monitor_index`, `window_scale`, `integer_scaling`, `aspect_fit`, `output_filter`, `sharpness`, `gamma`, `black_level`, `white_level`, `safe_area`, `frame_limit`, `fps_counter`, `screen_border`, `scanlines`, `crt_curve`, and `reduce_flashing`. The explicit resolution row persists through `windowSizeIndex`, `windowWidth`, `windowHeight`, and `windowSize` for compatibility with earlier native settings files; legacy files with a saved window size but no scale field load as `Custom` so the manual resolution is not overwritten by `Fit`. Settings files before `render_quality_version: 2` are migrated to Linear output filtering once, avoiding the chunky nearest-neighbor look from the previous scale implementation while preserving later user changes. Older settings files are tolerated: missing fields get defaults, older fullscreen/window/output/MSAA keys are still read, invalid enum/numeric values are clamped, and malformed JSON falls back to defaults.

Scene code, gameplay, TV composition, logical scissors, and mouse hitboxes remain in the original 960x540 coordinate space. Both backends rasterize those authored coordinates into targets whose dimensions exactly match the selected resolution; scene color, post-processing, MSAA resolve, and camera/TV capture all use that size. The finished frame alone scales to the actual swapchain/client viewport, so window decorations or desktop constraints cannot silently lower internal quality. Display mode, monitor selection, integer scaling, letterbox/overscan, safe area, and nearest/linear filtering remain presentation concerns and are not baked into in-game photos.

FXAA, CRT filter, screen noise, sharpness, gamma, brightness, contrast, black level, white level, TV mask, scanlines, and CRT curve are final-frame effects. Raylib uses `native/assets/shaders/postprocess.fs`; Vulkan uses `native/assets/shaders/vulkan/present2d.frag`. `FramePipeline::EndLogicalDraw()` supplies the same settings only during final presentation. `Reduce Flashing` scales optional static/noise/scanline intensity without removing gameplay information.

MSAA is selectable as Off, 2x, 4x, 8x, or 16x. Raylib uses a custom OpenGL multisample renderbuffer and resolve; Vulkan uses multisampled logical/gameplay images plus single-sample resolve attachments before post-processing, sampling, or readback. Each backend falls back to the highest supported level below the request and reports it. Vulkan applies a changed sample count on the next launch when live gameplay render targets already exist. Older settings containing `msaa4x` still load as 4x or Off.

Display mode options are Windowed, Borderless, and Fullscreen. The shared `Game` applies them through `AppWindow`; raylib maps them to its GLFW window calls and Vulkan maps them to Win32 styles, monitor placement, and swapchain recreation. VSync is also backend-specific behind the same shared settings flow. Driver/window-manager behavior can still vary by platform, while saved settings remain safe on the next startup.

Window Scale is based on 960x540: 1x, 2x, 3x, 4x, or Fit. The selected scale is clamped to the selected monitor while preserving aspect ratio. Window Size lists common sizes plus detected system resolutions; selecting one switches Window Scale to Custom and preserves the manual resolution until a fixed scale is chosen again. Integer Scaling affects final presentation scale only; if the window is smaller than 960x540, fractional downscale is allowed so the image remains usable. Aspect Fit defaults to Letterbox; Overscan fills while preserving aspect ratio and can crop the edges. Safe Area scales the final image inward and mouse conversion follows the adjusted viewport. Output Filter controls texture smoothing for loaded art/fonts and the final render target; new/reset settings default to Linear for smoother hand-drawn art at larger window scales.

Native HDR is available in the Windows x64 Vulkan build. The Display tab exposes Off, Auto, and On only when the renderer reports HDR-capable surface formats. Vulkan prefers scRGB, falls back to HDR10/PQ, submits metadata, and uses an SDR swapchain when HDR is unavailable or disabled. Raylib/OpenGL and 32-bit/Linux builds remain SDR.

Controller settings are intentionally omitted because the current native menu/gameplay input path is mouse and keyboard only. The settings scene does not embed web UI, Electron, Chromium, a webview, or JavaScript; it is drawn and updated entirely in C++/raylib.

## Native Localization Checklist

Localization data lives in `native/assets/lang/*.json` and is loaded by `native/src/Localization.cpp` through nlohmann_json. `Game` loads the saved `language_id`, falls back to English if the saved language is unavailable, and the TV-style settings page exposes a `Language` row in the Display tab. Missing localization keys fall back to English and log once per key/language pair.

Fan translation drop-ins live in external `translations/*.json` files and are loaded after official bundled language files. Runtime search locations are `translations/` beside the executable, `translations/` in the current working directory, and the detected asset/repository root `translations/` folder for development runs. The game creates those folders when possible, but never executes translation files or lets them request assets, scripts, shaders, sounds, or paths. Adding or removing fan translations while the game is already running currently requires a restart.

Invalid files are skipped with one warning; invalid entries inside otherwise valid fan files are counted in one warning. Missing keys fall back to English. ## Known Differences

- The original browser share/external buttons used `window.open`; the native compatibility layer asks the operating system to open the same repository links in the default browser.
- The bundled `css/Cairo-Regular.ttf` in this repository is a localized subset that does not render English reliably. The native port uses common OS Latin fonts on Windows/Linux, loads Latin-1 and Latin Extended-A glyphs for exposed Latin-script language packs, and falls back to raylib's default font only if none are available. Fan translations outside that range are data-safe but may need future font/layout work before every glyph renders correctly.
- Some PIXI/TweenJS container nesting details are flattened into backend-neutral renderer quads and state updates. The visible timings, scene order, screenshot capture, TV display, and gameplay progression are preserved.
- The original translated JS string tables remain in `js/textStrings.js` as inactive reference data. The browser fallback now defaults to `textStrings_EN`.

## Native Source Map

- `native/src/Scene.*`, `Scenes.*` -> preloader, quote, credits, post-credits, final scenes

- `native/src/Localization.*`, `native/assets/lang/*.json` -> native language packs and runtime string lookup
- `native/src/Gameplay.cpp` -> world systems, camera, TV, director, effects, peeps, staged gameplay
- `native/src/AssetManager.*` -> renderer texture/font handles and atlas JSON metadata
- `native/src/AudioManager.*`, `native/src/render/raylib/RaylibAudioBackend.*`, `native/src/render/miniaudio/*` -> backend-neutral sound/music control plus raylib and miniaudio playback
- `native/src/render/vulkan/*` -> Win32 x64 Vulkan app/input/capture services, renderer, presentation, HDR, and MSAA
- `native/src/Tween.*` -> timing/easing helpers
- `native/src/Text.h` -> localization key constants for scripted captions and scene overlays
- `native/resources/` and `sprites/app_icon.png` -> native executable/window/install icon assets
